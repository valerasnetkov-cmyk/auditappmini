# Do Not Do

## Не делать сейчас

- Не внедрять полный SaaS до стабилизации MVP.
- Не делать биллинг и тарифы на этом этапе.
- Не делать кастомные домены клиентов.
- Не делать управление ремонтом, задачи исполнителям и SLA.
- Не добавлять сложную иерархию холдинг / филиал / объект / участок.

---

## Номера техники

- Не возвращать QR как основной сценарий.
- Не разрешать кириллицу в ручном вводе номера.
- Не смешивать кириллицу и латиницу в `number_normalized`.
- Не запускать осмотр автоматически после OCR номера.

---

## Одометр

- Не сохранять распознанный километраж без подтверждения инспектора.
- Не убирать ручной ввод километража.
- Не завершать quick/planned осмотр без фото одометра.
- Не удалять фото одометра после распознавания.

---

## ДТП

- Не считать общий план повреждения достаточным доказательством.
- Не завершать ДТП-осмотр без места ДТП.
- Не завершать ДТП-осмотр без даты и времени ДТП.
- Не подставлять время создания осмотра как время ДТП без подтверждения инспектора.
- Не считать геоданные фото полноценной заменой текстового места ДТП.
- Не завершать ДТП-осмотр без крупного плана повреждённого участка.
- Не складывать все повреждения в одно фото, если повреждённых зон несколько.

---

## Фото

- Не принимать фото из галереи как доказательное фото.
- Не хранить фото в GitHub.
- Не делать публичные фото без проверки доступа.
- Не смешивать фото разных компаний в одном пути без `company_id`.

---

## Web / CSS

- Не использовать inline-стили `style={{ ... }}`.
- Не хардкодить цвета клиента в компонентах.
- Не хранить реальные ссылки, ключи или tenant-specific данные в CSS.
- Не хардкодить цвета тёмной темы внутри компонентов.
- Не делать отдельные компоненты под light и dark режимы.
- Не реализовывать тёмную тему только для части web-интерфейса.

---

## GitHub

- Не коммитить `.env`, ключи, токены, production URL, базы данных, загрузки, логи и реальные фото.

## Плановый осмотр

- Не превращать плановый осмотр в ремонтную диагностику.
- Не требовать разборки ходовой, тормозной системы или электрики.
- Не смешивать все дефекты в одну категорию `defect` без `component_area`.
- Не требовать фото каждого технического узла без необходимости, если это замедляет осмотр.
- Не завершать пункт с ответом `НЕТ` без комментария и фото, если дефект можно визуально зафиксировать.

## Mobile Theme

- Не хардкодить цвета на mobile-экранах.
- Не оставлять часть экранов только в светлой теме.
- Не делать тёмную тему, которая ухудшает читаемость фото, номера машины, одометра и ошибок.

## Не делать для региональной архитектуры

```txt
- не хранить всех клиентов в одной глобальной базе;
- не хранить РФ-компании в международном контуре, если требуется РФ-локализация;
- не отправлять фото РФ-компаний во внешний OCR вне региона;
- не использовать общий storage bucket для всех регионов;
- не смешивать backups разных регионов;
- не хранить персональные данные в global tenant registry;
- не менять регион компании простым UPDATE region_code;
- не делать отдельную ветку кода под каждый регион;
- не хранить реальные production-домены и endpoints в GitHub.
```

## Единицы измерения пробега

Не делать:

- не хранить значение одометра без единицы;
- не разрешать инспектору выбирать `km` / `mi` в каждом осмотре;
- не менять старые осмотры без сохранения исходной единицы;
- не показывать пробег без подписи `км` или `mi`;
- не считать OCR-единицу финальной без подтверждения инспектора;
- не сравнивать пробег `km` и `mi` напрямую без нормализации.

---

## Не делать при распознавании номера

```txt
- не начинать осмотр автоматически после OCR;
- не принимать OCR-результат без подтверждения инспектора;
- не хранить финальный номер кириллицей;
- не передавать OCR API tokens в mobile/web;
- не отправлять фото РФ-компаний в зарубежный OCR;
- не использовать один общий OCR endpoint без региональной проверки;
- не сохранять только raw_text без normalized_number;
- не блокировать ручной ввод, если OCR недоступен.
```
