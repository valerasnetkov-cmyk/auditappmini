# Vehicle Number Format

## Назначение файла

Этот файл описывает правила ручного ввода, нормализации, хранения и поиска номера машины.

QR-коды в проекте больше не используются. Инспектор вводит номер машины вручную.

---

## Главное правило ручного ввода

Ручной ввод номера разрешён **только латинскими буквами** и цифрами.

Для российских автомобильных номеров используются латинские визуальные аналоги кириллических букв:

```txt
A B E K M H O P C T Y X
```

Кириллица в поле ручного ввода не принимается.

Пример:

```txt
Правильно:   A123BC77
Неправильно: А123ВС77
```

Если инспектор видит на автомобиле номер `А123ВС77`, он вводит его как `A123BC77`.

---

## Главный принцип хранения

В базе нужно хранить два значения:

```txt
number_raw
number_normalized
```

Где:

- `number_raw` — номер, введённый пользователем в латинице;
- `number_normalized` — номер в едином латинском формате для поиска, сравнения и уникальности.

Все основные поля номера техники должны храниться латинскими символами.

---

## Формат хранения

Для поиска и уникальности использовать только `number_normalized`.

Правила нормализации ручного ввода:

- привести к верхнему регистру;
- удалить пробелы;
- удалить дефисы;
- удалить лишние разделители;
- проверить, что остались только цифры и латинские буквы из разрешённого набора;
- отклонить кириллицу и любые недопустимые символы.

---

## Российские автомобильные номера

На автомобильных номерах России используются только 12 букв кириллицы, имеющие визуальные аналоги в латинском алфавите:

```txt
А В Е К М Н О Р С Т У Х
```

В интерфейсе проекта инспектор вводит их **латинскими аналогами**:

```txt
A B E K M H O P C T Y X
```

Важно: это не транслитерация по звучанию, а визуальные аналоги.

---

## Примеры ручного ввода

```txt
A123BC77   -> корректно
M777OH199  -> корректно
X001YX97   -> корректно
А123ВС77   -> ошибка: используйте A123BC77
М777ОН199  -> ошибка: используйте M777OH199
```

---

## Примеры нормализации

```txt
A123BC77     -> A123BC77
A 123 BC 77  -> A123BC77
m777oh199    -> M777OH199
x001-yx-97   -> X001YX97
```

Кириллические примеры могут появляться только как результат OCR / ANPR или при миграции старых данных. В таком случае они приводятся к латинскому виду до показа инспектору:

```txt
А123ВС77  -> A123BC77
М777ОН199 -> M777OH199
Х001УХ97  -> X001YX97
```

---

## Что делать с кириллицей в ручном вводе

Если пользователь вручную вводит или вставляет кириллические символы, система должна показать ошибку.

Пример ошибки:

```txt
Введите номер латинскими буквами. Для российских номеров используйте A B E K M H O P C T Y X и цифры.
```

Допустимое поведение для UX: подсветить поле и показать подсказку с примером `A123BC77`.

Автоматически сохранять номер, введённый кириллицей, нельзя.

---

## Что делать с недопустимыми латинскими буквами

Если пользователь вводит латинские буквы, которых не может быть в российском номере, система должна показать ошибку.

Примеры недопустимых латинских букв для российского номера:

```txt
D F G I J L N Q R S V W Z
```

Пример ошибки:

```txt
Номер содержит недопустимые символы. Используйте только буквы A B E K M H O P C T Y X и цифры.
```

---

## Дипломатические номера

Дипломатические номера могут иметь отдельные особенности формата.

Для MVP:

```txt
дипломатические номера не обрабатываются отдельной логикой
```

Их поддержку можно добавить позже отдельным правилом валидации.

---

## Валидация MVP

Для MVP достаточно мягкой валидации:

- номер не пустой;
- номер введён латинскими буквами и цифрами;
- буквы входят в разрешённый набор `A B E K M H O P C T Y X`;
- длина находится в разумных пределах;
- поиск выполняется по `number_normalized`.

---

## Уникальность

Номер должен быть уникален внутри компании:

```txt
unique(company_id, number_normalized)
```

Один и тот же номер теоретически может существовать в разных компаниях, поэтому глобальная уникальность не нужна.

---

## Где использовать проверку номера

Проверка должна применяться:

- при создании техники;
- при обновлении номера техники;
- при поиске техники инспектором;
- при импорте техники;
- при подтверждении результата OCR;
- при проверке дублей;
- при отображении подсказок пользователю.

---

## Что не делать

- Не использовать QR-код как основной способ выбора техники.
- Не разрешать ручной ввод кириллицей.
- Не хранить кириллицу в `number_raw` и `number_normalized`.
- Не хранить только `number_raw`.
- Не искать технику по номеру без нормализации.
- Не смешивать кириллицу и латиницу в `number_normalized`.
- Не делать номер уникальным глобально для всех компаний.

---

## Использование при распознавании номера

Распознавание номера по фото должно использовать эти же правила финального формата.

Поток после OCR / ANPR:

```txt
raw_text -> normalizeVehicleNumber -> validateVehicleNumber -> показать инспектору в латинице -> подтверждение
```

OCR может вернуть кириллический текст, но экран подтверждения должен показывать и принимать только латинский номер.

Даже если номер распознан автоматически, система должна сохранять финальный номер только после подтверждения или исправления инспектором.

Подробный сценарий распознавания: `docs/vehicle-number-recognition.md`.

---

## Использование в новых типах осмотра

Правила номера одинаковы для всех типов осмотра: быстрый, плановый и ДТП.

Перед началом любого осмотра номер должен быть введён или распознан, приведён к латинице и подтверждён инспектором.
