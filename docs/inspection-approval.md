# Согласование завершённого осмотра

## Модель статусов

Согласование хранится отдельно от `completed`:

- `draft` — завершённый акт ещё не отправлен руководителю;
- `submitted` — акт ожидает решения;
- `approved` — акт принят;
- `rejected` — акт отклонён;
- `revision_required` — требуется новая доказательная фиксация.

`completed` продолжает означать техническую полноту. Завершённый осмотр,
фотографии, чек-лист и дефекты не разблокируются при возврате. Если нужны
другие материалы, инспектор создаёт новый осмотр, сохраняя исходный акт
неизменным.

## Переходы и роли

- Инспектор может отправить только собственный завершённый осмотр.
- Manager и owner могут отправлять завершённые осмотры компании.
- Решение `approved`, `rejected` или `revision_required` доступно только
  manager и owner.
- Отказ и возврат требуют комментарий.
- Повторная отправка разрешена после `rejected` и `revision_required`.
- Все переходы проверяют `company_id` и сохраняются в
  `inspection_approval_history`.

## API

- `GET /api/inspections/:id/approval` — текущий статус и история;
- `POST /api/inspections/:id/submit` — отправка на согласование;
- `POST /api/inspections/:id/approval` — решение руководителя.

Отправка создаёт адресные in-app уведомления manager и owner. Решение создаёт
уведомление инспектору. Они выдаются текущему пользователю через существующий
tenant-контракт `/api/company/usage`.

PDF-отчёт содержит текущий статус согласования, дату и комментарий решения.
