# Measurement Units / Единицы измерения пробега

## Цель

В системе должна быть настройка единиц измерения пробега для каждой компании: километры или мили.

Настройка влияет на:

- мобильное приложение инспектора;
- web-панель руководителя и администратора;
- карточку техники;
- карточку осмотра;
- историю пробега;
- OCR / распознавание одометра;
- отчёты и экспорт.

---

## Поддерживаемые единицы

```txt
km -> километры
mi -> мили
```

По умолчанию:

```txt
RU-контур   -> km
EU-контур   -> km
INTL-контур -> km или mi в зависимости от страны компании
```

Для MVP можно оставить глобальное значение по умолчанию `km`, но обязательно дать администратору компании возможность изменить его в настройках.

---

## Где настраивается

В web-панели администратора компании:

```txt
/settings/company
  -> Единицы измерения
     Пробег: [Километры / Мили]
```

Доступ к изменению настройки:

```txt
owner
admin
```

`manager` и `inspector` не должны менять единицы измерения компании.

---

## Основное правило хранения

Чтобы избежать ошибок при смене настройки, система должна хранить и значение, и единицу измерения.

```txt
odometer_value
odometer_unit
```

Пример:

```txt
128450 km
79815 mi
```

Нельзя хранить только число без единицы измерения.

---

## Рекомендованное внутреннее хранение

Для аналитики и сравнения пробега лучше дополнительно хранить нормализованное значение в километрах:

```txt
odometer_value        -> значение, которое подтвердил инспектор
odometer_unit         -> единица, выбранная компанией на момент осмотра: km / mi
odometer_value_km     -> нормализованное значение в километрах для сортировки и аналитики
```

Пример:

```txt
odometer_value: 10000
odometer_unit: mi
odometer_value_km: 16093
```

`odometer_value_km` не показывается пользователю, но используется для:

- проверки уменьшения пробега;
- графиков;
- фильтров;
- отчётов;
- будущей аналитики.

---

## Конвертация

Формула:

```txt
1 mi = 1.609344 km
1 km = 0.621371192237334 mi
```

Правило округления для интерфейса:

```txt
пробег автомобиля -> целое число
```

Пример:

```txt
10000 mi -> 16093 km
16093 km -> 9999 mi или 10000 mi после округления
```

В отчётах лучше показывать только выбранную единицу компании, чтобы не путать пользователей.

---

## Поведение при смене настройки компании

Если администратор меняет `distance_unit` с `km` на `mi` или наоборот:

1. Старые осмотры не должны терять исходную единицу.
2. История должна показываться в текущей единице компании, если включён режим отображения `current_company_unit`.
3. В деталях осмотра желательно показывать исходную единицу на момент осмотра.

Пример:

```txt
Компания раньше использовала km.
Осмотр был сохранён как 128450 km.
Позже компания переключилась на mi.

Карточка техники может показать: 79815 mi
Карточка осмотра может показать: 128450 km, исходная единица осмотра
```

Для MVP проще:

```txt
старые осмотры показывать в той единице, в которой они были сохранены;
новые осмотры сохранять в новой единице компании.
```

## Мобильное приложение

На экране одометра мобильное приложение должно показывать единицу измерения компании:

```txt
Введите пробег, км
```

или:

```txt
Введите пробег, mi
```

Правила:

- инспектор не выбирает единицу измерения вручную в каждом осмотре;
- единица подтягивается из настроек компании;
- при offline-осмотре единица должна быть сохранена в локальном draft;
- при синхронизации отправляется и значение, и единица.

Запрос сохранения:

```json
{
  "odometerValue": 128450,
  "odometerUnit": "km",
  "photoId": "photo-id",
  "recognitionId": "recognition-id"
}
```

## OCR / распознавание одометра

OCR распознаёт число, но финальная единица берётся из настроек компании.

```txt
Фото одометра -> OCR число -> единица компании -> подтверждение инспектором
```

OCR не должен сам решать, километры это или мили, если на фото нет однозначной маркировки.

Если на фото явно видна единица, отличающаяся от настройки компании, система должна показать предупреждение:

```txt
На фото возможно значение в милях, а компания использует километры. Проверьте показание.
```

Для MVP это предупреждение можно не блокировать, но нужно дать инспектору исправить значение вручную.

## Web-интерфейс

В web нужно отображать единицу рядом с каждым значением пробега:

```txt
Пробег: 128450 км
Пробег: 79815 mi
```

В таблицах:

| Поле | Пример |
|---|---|
| Последний пробег | 128450 км |
| Источник | OCR + подтверждение |
| Фото одометра | открыть |

В настройках компании:

```txt
Единицы измерения пробега:
(•) Километры
( ) Мили
```

При изменении настройки показать предупреждение:

```txt
Новые осмотры будут использовать выбранную единицу измерения. Старые записи сохранят исходную единицу.
```

## Backend

Backend должен:

- возвращать `distance_unit` в настройках компании;
- принимать `odometerUnit` при сохранении одометра;
- валидировать только `km` или `mi`;
- сохранять исходное значение и единицу;
- рассчитывать `odometer_value_km` для внутренних проверок;
- проверять уменьшение пробега в нормализованном виде.

Endpoint настройки компании:

```http
PATCH /api/company/settings
```

Пример запроса:

```json
{
  "distanceUnit": "mi"
}
```

Пример ответа:

```json
{
  "companyId": "company-id",
  "distanceUnit": "mi"
}
```

Добавить в `companies`:

```txt
distance_unit
```

```txt

```

```txt

```

## Что не делать

Не делать:

- не хранить пробег без единицы измерения;
- не разрешать инспектору менять единицу в каждом осмотре;
- не конвертировать старые записи без сохранения исходной единицы;
- не смешивать `km` и `mi` в одной таблице без явного поля `odometer_unit`;
- не показывать значение пробега без подписи `км` или `mi`;
- не считать OCR-единицу финальной без подтверждения инспектора.
