# Mobile Changes

## Цель

Мобильное приложение должно максимально быстро проводить инспектора через осмотр, но при этом собирать доказательные данные: номер, обязательные фото, одометр, чек-лист, дефекты и геоданные.

---

## Новый основной сценарий

```txt
Вход -> Выбор компании -> Начать осмотр -> Ввод номера -> Подтверждение техники -> Выбор типа осмотра -> Фото -> Одометр -> Чек-лист -> Дефекты -> Завершение
```

QR-коды не используются.

---

## Экран ввода номера машины

Основной сценарий — ручной ввод.

Правила:

- только латинские буквы и цифры;
- кириллица не принимается;
- пробелы и дефисы можно автоматически удалять;
- для российских номеров используются `A B E K M H O P C T Y X`.

Дополнительная кнопка:

```txt
[Распознать номер по фото]
```

Распознанный номер должен быть подтверждён или исправлен инспектором.

---

## Экран выбора типа осмотра

Доступные типы:

```txt
[Быстрый]
[Плановый]
[ДТП]
```

После выбора типа приложение загружает список обязательных фото.

---

## Экран обязательных фото

Приложение показывает чек-лист фото:

### Быстрый

```txt
[ ] Фронтальная часть
[ ] Левый борт
[ ] Правый борт
[ ] Задняя часть
[ ] Общий план
[ ] Одометр
```

### Плановый

```txt
[ ] Фронтальная часть
[ ] Левый борт
[ ] Правый борт
[ ] Задняя часть
[ ] Общий план
[ ] Одометр
[ ] Дополнительные фото при необходимости
```

### ДТП

```txt
[ ] Общий план автомобиля
[ ] Фронтальная часть
[ ] Левый борт
[ ] Правый борт
[ ] Задняя часть
[ ] Общий план повреждения
[ ] Крупный план повреждённого участка
[ ] Одометр, если доступно
```

Если повреждений несколько, инспектор добавляет несколько фото `Крупный план повреждённого участка`.

---

## Экран одометра

Сценарий:

1. Сделать фото одометра.
2. Запустить распознавание километража.
3. Показать результат.
4. Дать инспектору подтвердить или исправить значение.
5. Сохранить значение в карточку осмотра.

Пример:

```txt
Распознан километраж: 128450 км

[Подтвердить]
[Исправить]
[Сделать фото заново]
```

Ручной ввод обязателен как fallback.

Правила ручного ввода:

- только цифры;
- значение не может быть отрицательным;
- если новое значение меньше предыдущего, показать предупреждение.

---

## Экран чек-листа

Для каждого пункта:

```txt
[Пункт проверки]
[ДА] [НЕТ]
Комментарий
```

Если выбран `НЕТ`:

- создаётся дефект;
- требуется фото дефекта;
- без фото дефекта завершить осмотр нельзя.

---

## Осмотр ДТП

Для ДТП нужен отдельный UX-блок:

```txt
Повреждённый участок
[Описание повреждения]
[Фото общего плана]
[Фото крупного плана]
[Добавить ещё повреждение]
```

Перед блоком повреждений показывается обязательный блок данных ДТП:

```txt
Данные ДТП
[Дата и время ДТП]
[Место ДТП]
[Определить координаты]
[Подтвердить данные ДТП]
```

Дата/время ДТП и место ДТП обязательны. Если координаты определены автоматически, инспектор всё равно подтверждает или исправляет текстовое место.

Правило:

- дата и время ДТП обязательны;
- место ДТП обязательно;
- общий план обязателен;
- close-up обязателен;
- при нескольких повреждениях каждое повреждение фотографируется отдельно.

---

## Завершение осмотра

Перед кнопкой `Завершить` приложение показывает готовность:

```txt
Номер подтверждён: да
Обязательные фото: 6/6
Одометр: 128450 км
Чек-лист заполнен: да
Дефекты с фото: да
Данные ДТП: заполнены, если тип ДТП
``` 

Если данные неполные, кнопка завершения заблокирована или backend возвращает список ошибок.

---

## Оффлайн-очередь

Мобильное приложение должно уметь:

- сохранять черновик осмотра локально;
- хранить фото локально до отправки;
- показывать статус синхронизации;
- повторять отправку при появлении сети;
- не терять подтверждённый номер и километраж.

---

## Мультиязычность mobile

Поддерживаемые языки:

```txt
ru
en
de
fr
es
```

Переводятся кнопки, подсказки, типы осмотров, типы фото, ошибки и статусы.

Пользовательские данные не переводятся.

---

## Что удалить из mobile-сценария

- QR-сканирование как основной способ выбора техники.
- Запуск осмотра без подтверждённого номера.
- Завершение осмотра без фото одометра для quick/planned.
- Завершение ДТП-осмотра без места и времени ДТП.
- Завершение ДТП-осмотра без close-up повреждения.
- Автоматическое принятие OCR-результата без подтверждения инспектора.

## Плановый осмотр: технические блоки

В плановом осмотре после обязательных внешних фото и одометра приложение показывает расширенный чек-лист по секциям:

```txt
1. Внешнее состояние
2. Ходовая часть
3. Тормозная система
4. Электрика
5. Световые приборы
```

UX для секции:

```txt
[Название секции]
[Пункт проверки]
[ДА] [НЕТ] [Н/П]
Комментарий
Фото при отклонении
```

`Н/П` означает `не применимо` или `невозможно проверить`, если пункт не относится к конкретной технике или нет доступа без разборки.

Примеры пунктов:

### Ходовая часть

- нет видимых повреждений подвески;
- нет течей амортизаторов;
- шины и колёса без критичных повреждений;
- нет повреждений пыльников и защитных элементов.

### Тормозная система

- нет видимых подтёков тормозной жидкости;
- тормозные шланги/трубки без видимых повреждений;
- нет предупреждений тормозной системы на панели приборов;
- стояночный тормоз работает, если проверка предусмотрена.

### Электрика

- аккумулятор и клеммы без видимых повреждений, если доступны;
- видимая проводка без повреждений;
- приборная панель без критичных ошибок;
- стеклоочистители/омыватель работают, если включены в регламент.

### Световые приборы

- ближний свет работает;
- дальний свет работает;
- габариты работают;
- стоп-сигналы работают;
- поворотники работают;
- аварийная сигнализация работает;
- фонари заднего хода работают;
- подсветка номера работает.

Если выбран `НЕТ`, приложение требует комментарий и фото дефекта, если его можно зафиксировать визуально.

---

## Тёмная тема mobile

Мобильное приложение должно поддерживать три режима темы:

```txt
system
light
dark
```

Тема должна применяться ко всем экранам:

- вход;
- выбор компании;
- ввод номера машины;
- OCR номера;
- выбор типа осмотра;
- обязательные фото;
- одометр и OCR километража;
- чек-лист;
- фиксация ДТП;
- завершение осмотра;
- оффлайн-очередь.

В настройках пользователя добавить переключатель:

```txt
Тема интерфейса
- Как в системе
- Светлая
- Тёмная
```

Технические правила:

- не хардкодить цвета на экранах;
- использовать единый theme provider / theme tokens;
- учитывать системную тему устройства для режима `system`;
- тема должна сохраняться между сессиями;
- превью фото, номер машины, показания одометра и ошибки должны быть читаемы в обеих темах.

Минимальные mobile theme tokens:

```txt
background
surface
card
text
mutedText
border
primary
danger
warning
success
inputBackground
buttonText
```

Тёмная тема не должна менять бизнес-логику осмотра, требования к фото, OCR или валидацию.

## Regional mobile behavior

Мобильное приложение должно выбирать API региона автоматически.

После логина пользователь получает список доступных компаний:

```json
[
  {
    "companyId": "company-id",
    "name": "Company name",
    "role": "inspector",
    "regionCode": "ru",
    "apiClusterKey": "ru-api"
  }
]
```

Если компания одна — приложение выбирает её автоматически. Если компаний несколько — пользователь выбирает компанию.

Все фото, OCR номера, OCR одометра и синхронизация offline-очереди отправляются только в API региона выбранной компании.

Offline-очередь должна хранить `company_id` и `region_code`, чтобы не отправить осмотр в неправильный регион после повторного входа.

## Единицы измерения пробега

Мобильное приложение получает единицу пробега из настроек компании:

```txt
company.distance_unit = km | mi
```

На экране одометра показывать:

```txt
Введите пробег, км
```

или:

```txt
Введите пробег, mi
```

Правила:

- инспектор не выбирает единицу измерения в каждом осмотре;
- при offline-осмотре единица сохраняется в draft;
- при синхронизации отправляются `odometerValue` и `odometerUnit`;
- если OCR распознал только число, единица берётся из настроек компании;
- если на фото видна другая единица, приложение показывает предупреждение и предлагает проверить значение.

## Распознавание номера по фото

На экране ввода номера добавляется кнопка:

```txt
[Распознать номер по фото]
```

Поток:

```txt
Открыть камеру
-> сделать фото номера
-> отправить фото на /api/vehicle-number/recognize
-> получить candidates
-> показать лучший вариант
-> инспектор подтверждает или исправляет номер
-> вызвать /api/vehicles/resolve-number
```

UI-состояния:

```txt
Распознан номер:
A123BC77

[Подтвердить]
[Исправить]
[Сделать фото заново]
```

Если номер не найден:

```txt
Номер не распознан.
Введите номер вручную латинскими буквами.
```

Правила:

- ручной ввод остаётся доступным всегда;
- кириллица в ручном вводе запрещена;
- OCR-результат можно подтвердить только после показа инспектору;
- низкая уверенность распознавания не блокирует ручной ввод;
- если устройство offline, кнопка распознавания может быть недоступна, но ручной ввод должен работать;
- осмотр нельзя начать, пока номер не подтверждён.

Пример mobile upload:

```ts
const photo = await cameraRef.current?.takePictureAsync({
  quality: 0.8,
  exif: true
});

const formData = new FormData();

formData.append("image", {
  uri: photo.uri,
  name: "plate.jpg",
  type: "image/jpeg"
} as any);

const response = await fetch(`${API_URL}/vehicle-number/recognize`, {
  method: "POST",
  headers: {
    Authorization: `Bearer ${token}`
  },
  body: formData
});

const result = await response.json();
```
