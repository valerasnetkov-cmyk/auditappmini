# Product

## Коммерческая модель

AuditAvto монетизируется по размеру автопарка и уровню контроля: техника,
пользователи, осмотры, фото-хранилище, OCR и доступные модули. Базовые тарифы
и правила оплаты описаны в [billing-and-tariffs.md](./billing-and-tariffs.md).

## Назначение

Проект фиксирует состояние техники с фото-доказательствами, историей осмотров, дефектами, пробегом и отдельным сценарием ДТП-фиксации.

Продукт не управляет ремонтом, не назначает исполнителей и не контролирует SLA. Основная ценность — доказательно зафиксировать состояние техники на момент осмотра.

---

## Основной процесс

```txt
Ввод номера -> Подтверждение техники -> Выбор типа осмотра -> Фото -> Чек-лист -> Дефекты -> Одометр -> Завершение -> История
```

---

## Типы осмотра

### Быстрый осмотр

Основной регулярный сценарий. Цель — быстро зафиксировать текущее состояние техники.

Обязательные элементы:

- подтверждённый номер техники;
- фото фронтальной части;
- фото левого борта;
- фото правого борта;
- фото задней части;
- общий план;
- фото одометра;
- подтверждённый километраж;
- короткий чек-лист.

### Плановый осмотр

Более полный сценарий для регулярной проверки по расписанию или внутреннему регламенту.

Обязательные элементы:

- все фото быстрого осмотра;
- дополнительные общие планы при необходимости;
- расширенный чек-лист;
- комментарии инспектора по отклонениям;
- фото одометра;
- подтверждённый километраж.

### Осмотр ДТП

Сценарий фиксации последствий ДТП или инцидента.

Обязательные элементы:

- общий план автомобиля;
- фото фронтальной части;
- фото левого борта;
- фото правого борта;
- фото задней части;
- фото повреждённого участка крупным планом;
- отдельные крупные планы для каждого повреждённого участка;
- описание повреждения;
- место ДТП;
- дата и время ДТП;
- координаты ДТП, если доступны;
- фото одометра, если доступно;
- километраж или причина невозможности фиксации.

---

## Правила фотофиксации

- Фото делаются только через камеру.
- Фото из галереи не используются для доказательной фиксации.
- Каждое фото сохраняет тип, дату, время и геоданные.
- Осмотр нельзя завершить без обязательных фото выбранного типа осмотра.
- Для ДТП общий план не заменяет крупный план повреждения.
- Если повреждений несколько, каждое повреждение фиксируется отдельным близким фото.

---

## Место и время ДТП

Для осмотра ДТП обязательно фиксируются:

```txt
дата и время ДТП
место ДТП текстом
координаты ДТП, если доступны
```

Время создания осмотра не заменяет время ДТП. Геоданные фото не заменяют текстовое место ДТП. Инспектор должен подтвердить или исправить автоматически определённые данные перед завершением осмотра.

---

## Одометр и километраж

Фото одометра является обязательным для быстрого и планового осмотра.

Для ДТП фото одометра желательно и обязательно, если доступ к салону возможен. Если доступа нет, инспектор выбирает причину недоступности.

Распознавание километража работает как помощник:

```txt
Фото одометра -> распознавание -> результат -> подтверждение или исправление инспектором -> запись в карточку осмотра
```

Километраж не считается финальным без подтверждения инспектора. Ручной ввод должен быть доступен всегда.

---

## Номер техники

- QR-коды не используются.
- Основной сценарий — ручной ввод номера.
- Ввод номера только латиницей и цифрами.
- Для российских номеров используются визуальные латинские аналоги: `A B E K M H O P C T Y X`.
- Распознавание номера по фото — вспомогательная функция.
- Распознанный номер должен быть подтверждён или исправлен инспектором.

## Роли

### Inspector

Проводит осмотр, вводит номер, подтверждает технику, делает фото, фиксирует километраж, отмечает чек-лист и дефекты.

### Manager

Просматривает историю техники, осмотры, дефекты, пробег, фото и статистику.

### Admin / Owner

Управляет компанией, пользователями и настройками. Нужен для SaaS-ready модели.

---

## SaaS-ready правила

- Все бизнес-данные имеют `company_id`.
- Пользователь видит только данные своей компании.
- Фото хранятся в структуре с `company_id`.
- Язык интерфейса хранится на уровне пользователя и компании.
- Пользовательские данные не переводятся автоматически.

---

## Что не входит в MVP

- управление ремонтом;
- задачи исполнителям;
- SLA;
- биллинг;
- публичная регистрация компаний;
- кастомные домены клиентов;
- сложная аналитика;
- автоматическое завершение осмотра на основе OCR без подтверждения инспектора.

## Плановый осмотр: технические блоки

Плановый осмотр должен учитывать не только внешние дефекты, но и доступные для проверки технические системы.

Обязательные блоки планового чек-листа:

```txt
exterior      -> внешнее состояние
undercarriage -> ходовая часть
brake_system  -> тормозная система
electrical    -> электрика
lighting      -> световые приборы
odometer      -> одометр / пробег
```

Световые приборы относятся к электрике, но в интерфейсе должны отображаться отдельной секцией, потому что это частая и понятная проверка для инспектора.

Если по любому пункту выбран ответ `НЕТ`, создаётся дефект с категорией технической системы. Фото дефекта обязательно, если дефект можно зафиксировать визуально. Если фото невозможно, инспектор указывает причину в комментарии.

Плановый осмотр не должен превращаться в ремонтную диагностику. Инспектор фиксирует доступные признаки неисправности, фото и комментарий, но не назначает ремонт и не ставит окончательный технический диагноз.

## Тема интерфейса

Система должна поддерживать светлую и тёмную тему для web/ПК и мобильного приложения.

Поддерживаемые режимы:

```txt
system
light
dark
```

Тёмная тема не меняет бизнес-логику осмотров. Она влияет только на визуальное отображение интерфейса.

Обязательные правила:

- пользователь может выбрать тему вручную;
- по умолчанию используется системная тема устройства;
- web и mobile должны иметь одинаковую логику выбора темы;
- номера машин, фото, одометр, ошибки и дефекты должны быть читаемы в обеих темах.

## Региональность продукта

Для multi-company SaaS каждая компания должна быть закреплена за регионом хранения данных.

```txt
RU company -> RU контур
EU company -> EU контур
INTL company -> INTL контур
```

Региональность не должна быть видимой сложностью для инспектора. Пользователь просто входит в компанию, а приложение автоматически использует нужный API, storage и OCR.

OCR номера и одометра работает только в регионе компании.

## Единицы измерения пробега

Компания может работать в километрах или милях.

```txt
km -> километры
mi -> мили
```

Единица выбирается владельцем компании в web-панели. Инспектор видит выбранную единицу на экране одометра, но не меняет её вручную во время осмотра.

Бизнес-правила:

- пробег нельзя сохранять без единицы измерения;
- OCR одометра распознаёт число, а единица берётся из настроек компании;
- пробег в карточке техники, карточке осмотра и отчётах всегда отображается с единицей;
- при смене настройки компании старые осмотры сохраняют исходную единицу;
- проверка уменьшения пробега должна выполняться в нормализованном виде.
