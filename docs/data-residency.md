# Data Residency and Regional Hosting

## Цель

Проект должен поддерживать несколько региональных контуров хранения данных, чтобы компании из разных стран могли работать в своём регионе.

Базовый принцип:

```txt
Один код -> несколько региональных деплоев -> каждая компания закреплена за одним регионом
```

---

## Юридическая оговорка

Этот документ описывает техническую архитектуру и продуктовые правила. Он не заменяет юридическую консультацию. Перед запуском в конкретной стране необходимо проверить требования по персональным данным, хранению фото, геоданным, данным ДТП и обработке изображений.

---

## Почему данные нужно разделять по регионам

В проекте хранятся потенциально чувствительные данные:

```txt
- пользователи и роли;
- номера автомобилей;
- фото автомобиля;
- фото номера;
- фото одометра;
- место и время ДТП;
- геоданные;
- комментарии инспектора;
- история осмотров;
- audit logs.
```

Поэтому данные компании не должны автоматически попадать в общий глобальный контур.

---

## Рекомендуемые контуры

Для MVP достаточно двух или трёх контуров:

```txt
ru     -> компании РФ / пользователи РФ
intl   -> международные компании вне РФ
```

Позже можно разделить международный контур:

```txt
ru     -> Россия
 eu    -> Европейский союз / EEA
intl   -> остальные страны
```

---

## RU-контур

Назначение:

```txt
Российские компании и пользователи, для которых требуется хранение данных на территории РФ.
```

Рекомендуемая инфраструктура:

```txt
- PostgreSQL на РФ-хостинге;
- S3-compatible storage на РФ-хостинге;
- backend API в РФ;
- OCR номера и одометра в РФ;
- backup внутри РФ;
- логи внутри РФ.
```

Возможные провайдеры для оценки:

```txt
- Yandex Cloud;
- Selectel;
- VK Cloud;
- другие РФ-провайдеры с нужной инфраструктурой и документами.
```

---

## EU-контур

Назначение:

```txt
Компании из EU / EEA, если проект выходит на европейский рынок.
```

Рекомендуемая инфраструктура:

```txt
- Supabase EU или PostgreSQL в EU;
- storage в EU;
- backend API в EU;
- OCR в EU;
- backup в EU.
```

Важно: перед передачей персональных данных из EU/EEA в третьи страны нужно отдельно проверять правовое основание и механизм передачи данных.

---

## INTL-контур

Назначение:

```txt
Компании вне РФ и EU, если нет отдельного локального требования по хранению.
```

Рекомендуемая инфраструктура:

```txt
- Supabase / PostgreSQL в выбранном регионе;
- storage в том же регионе;
- OCR в том же регионе;
- backup в том же регионе.
```

---

## Что можно хранить глобально

Глобально можно хранить только минимальный tenant registry для маршрутизации.

Разрешённые поля:

```txt
tenant_id
slug
public_name
region_code
status
created_at
```

Не хранить глобально:

```txt
email пользователей
ФИО
фото
номера автомобилей
геолокации
данные ДТП
комментарии
JWT secrets
storage keys
service keys
```

---

## Что хранится внутри регионального контура

```txt
companies
profiles
company_users
vehicles
inspections
inspection_items
defects
photos
photo_requirements
vehicle_number_recognitions
odometer_recognitions
audit_logs
```

## OCR и региональность

OCR номера и одометра должен выполняться в том же регионе, где находится компания.

Правильно:

```txt
RU company -> RU storage -> RU OCR -> RU DB
EU company -> EU storage -> EU OCR -> EU DB
INTL company -> INTL storage -> INTL OCR -> INTL DB
```

Неправильно:

```txt
RU company -> EU/US OCR
EU company -> RU OCR
общий OCR-сервис без регионального разделения
```

---

## Backup

Backup должен оставаться внутри того же региона:

```txt
ru-db -> ru-backup
 eu-db -> eu-backup
intl-db -> intl-backup
```

Не допускается общий backup bucket для всех регионов.

---

## Перенос компании между регионами

Перенос компании между регионами — отдельная миграция, а не простое изменение поля `region_code`.

Минимальный порядок:

```txt
1. Заблокировать изменения tenant на время миграции.
2. Экспортировать данные компании из старого региона.
3. Перенести фото и метаданные.
4. Проверить checksum / hash фото.
5. Импортировать данные в новый регион.
6. Обновить global tenant registry.
7. Проверить доступы пользователей.
8. Разблокировать tenant.
9. Сохранить audit log миграции.
```

## Источники для проверки актуальности

- Федеральный закон РФ № 152-ФЗ «О персональных данных», статья 18.
- European Commission: rules on international data transfers and Standard Contractual Clauses.
- Supabase Docs: available regions and project region migration.
- Yandex Cloud Docs: regions and availability zones.

## OCR regional routing

OCR/ANPR является частью регионального контура.

```txt
ru-api -> ru-ocr -> ru-storage -> ru-db
eu-api -> eu-ocr -> eu-storage -> eu-db
intl-api -> intl-ocr -> intl-storage -> intl-db
```

Запрещено отправлять фото номера, фото одометра, фото ДТП и другие доказательные материалы в OCR другого региона.

Backend должен возвращать `region_mismatch`, если OCR-провайдер не соответствует региону tenant.
