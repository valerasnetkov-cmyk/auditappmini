# Надёжный доказательный осмотр (P0)

## Статус осмотра

`completed` означает техническую полноту доказательств. После завершения
backend запрещает изменение осмотра, чек-листа, дефектов и фотографий с кодом
`INSPECTION_ALREADY_COMPLETED`. Будущее согласование должно использовать
отдельный `approval_status`.

## Идемпотентность и синхронизация

- `POST /api/inspections` принимает `client_inspection_id` и `sync_source`.
  Повтор с тем же client ID внутри tenant возвращает существующий осмотр.
- Загрузка фото принимает `client_photo_id`, `captured_at`, `captured_lat` и
  `captured_lng`. Повтор возвращает существующую запись и удаляет временный
  загруженный дубликат.
- Mobile хранит активный черновик в AsyncStorage, а локальные фото копирует в
  Expo document storage. Очередь загружает не более двух фото параллельно и
  возобновляется при старте, возврате приложения в active и вручную.
- Новый осмотр требует сети. Уже созданный осмотр продолжает работать без сети.
  Локальные файлы удаляются только после успешного завершения и полной
  синхронизации.

## Readiness

`GET /api/inspections/:id/readiness` возвращает единый контракт для mobile,
web и завершения:

```json
{
  "ready": false,
  "missing": [
    {
      "code": "missing_required_photo",
      "field": "rear",
      "label": "Фото сзади"
    }
  ]
}
```

Валидатор проверяет все пункты чек-листа, дефект для каждого отрицательного
ответа, фото каждого дефекта, обязательные категории фото и пробег. Для ДТП
также обязательны время и место, а пробег задаётся либо подтверждённым
значением с фото одометра, либо причиной недоступности.

`POST /api/inspections/:id/complete` при неполноте отвечает `400` с кодом
`INSPECTION_COMPLETION_BLOCKED` и тем же массивом `missing`.

## Watermark и PDF

Оригиналы фотографий не изменяются. Sharp создаёт отдельный `watermark.webp`;
для старых фото он генерируется при первом формировании отчёта.

Доступные авторизованные endpoint'ы:

- `POST /api/inspections/:id/report` — сформировать или пересоздать PDF;
- `GET /api/inspections/:id/report` — получить метаданные и SHA-256;
- `GET /api/inspections/:id/report.pdf` — скачать PDF.

Отчёт формируется сервером через PDFKit, использует локальный DejaVu Sans для
кириллицы и содержит QR на авторизованную страницу
`/inspections/:id/report`. Незавершённые осмотры и данные другого tenant
отклоняются.

PDF сначала формируется во временном файле и публикуется только после полного
завершения записи. Параллельные запросы генерации одного отчёта выполняются
последовательно. При чтении metadata и скачивании backend заново вычисляет
SHA-256 и проверяет размер файла. Расхождение переводит отчёт в состояние
`corrupted`, задаёт `integrity_status = mismatch` и блокирует скачивание с
кодом `REPORT_INTEGRITY_FAILED` до повторной генерации.
