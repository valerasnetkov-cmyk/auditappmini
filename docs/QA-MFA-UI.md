QA Plan: MFA in Web Admin and Mobile Flow

Goal
- Validate end-to-end MFA enablement for a user via the admin panel, including scanning QR, verifying TOTp, and token issuance to the user upon MFA activation.

Preconditions
- Backend with MFA endpoints: /api/users/:id/mfa/setup and /api/users/:id/mfa/verify exist and working.
- Admin user with permission to manage users.
- Admin panel MFA UI pages accessible at /admin/mfa and /admin/mfa/[id].
- A test user exists in the system (e.g. test inspector user) or use the demo data generated by /seed endpoints.
-
Test Scenarios
1. MFA Enablement Flow (Admin)
  a. Navigate to MFA list: /admin/mfa
  b. Choose a user and click “Настроить MFA”
  c. Backend returns otpauth_url and secret; QR code is displayed
  d. Admin uses a authenticator app to scan QR or enter secret
  e. Admin clicks a Verify button and inputs the TOTp generated by the authenticator
  f. Backend validates TOTp, MFA is enabled for user; subsequent logins by the user will require MFA
  g. Admin can show/hide the MFA secret on the per-user page and copy it to clipboard

2. User Login Flow (Mobile/Web) after MFA Enabled
  a. Login as that user via web/mobile with credentials
  b. Server returns mfaRequired true; user is redirected to MFA verification page (if applicable) or informed via a route
  c. User enters 6-digit TOTp into the MFA screen
  d. After successful verification, server issues a new JWT; user gains access to the app

3. Error Scenarios
  a. Wrong TOTp code should return 401 and show an error toast on UI
  b. MFA not configured but login attempted should be standard login without MFA code, unless configured as required
  c. MFA re-setup should overwrite previous secret for the user if admin configures again

4. Edge Cases
  a. Admin tries to configure MFA for a non-existent user → appropriate error
  b. Non-manager user attempts to configure MFA for others → 403 forbidden

5. UI Enhancements
  - Verify that on /admin/mfa/[id] the secret can be toggled visible/invisible and copied

Checklist (manual)
- [ ] MFA Setup: QR code displays after setup call
- [ ] TOTp Verification: valid code enables MFA and returns new JWT when applicable
- [ ] Admin page: MFA status reflects enabled/disabled per user
- [ ] MFA flow on login: once MFA is enabled for a user, login requires MFA
- [ ] Error handling: invalid codes return proper error messages and toasts
- [ ] Security: tokens stored securely on client and server validates MFA state

Notes
- This plan assumes MFA is implemented as a separate step after initial login, and that the admin panel can configure MFA per user.
- If a simpler MVP is preferred, we can implement a lighter version (admin sets MFA for one user, then test via that user).
