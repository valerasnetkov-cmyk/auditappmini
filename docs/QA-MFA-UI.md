QA Plan: MFA in Company Owner UI and Mobile Flow

Goal
- Validate end-to-end MFA enablement for a user via the company owner panel, including scanning QR, verifying TOTP, and token issuance to the user upon MFA activation. Resource admin does not configure MFA for tenant users.

Preconditions
- Backend with MFA endpoints: /api/users/:id/mfa/setup, /api/users/:id/mfa/enable, and /api/users/:id/mfa/disable exist and working. /api/users/:id/mfa/verify remains a backwards-compatible alias for enable.
- Company owner user with permission to manage users inside their company.
- Company owner MFA UI pages accessible at /admin/mfa and /admin/mfa/[id].
- A test user exists in the system (e.g. test inspector user) or use the demo data generated by /seed endpoints.
-
Test Scenarios
1. MFA Enablement Flow (Company Owner)
  a. Navigate to MFA list: /admin/mfa
  b. Choose a user and click “Настроить MFA”
  c. Backend returns otpauth_url and secret; QR code is displayed
  d. Company owner uses an authenticator app to scan QR or enter secret
  e. Company owner clicks a Verify button and inputs the TOTP generated by the authenticator
  f. Backend validates TOTp, MFA is enabled for user; subsequent logins by the user will require MFA
  g. Company owner can show/hide the MFA secret on the per-user page and copy it to clipboard

2. User Login Flow (Mobile/Web) after MFA Enabled
  a. Login as that user via web/mobile with credentials
  b. Server returns mfaRequired true; user is redirected to MFA verification page (if applicable) or informed via a route
  c. User enters 6-digit TOTp into the MFA screen
  d. After successful verification, server issues a new JWT; user gains access to the app

3. Error Scenarios
  a. Wrong TOTp code should return 401 and show an error toast on UI
  b. MFA not configured but login attempted should be standard login without MFA code, unless configured as required
  c. MFA re-setup should overwrite previous secret for the user if company owner configures again

4. Edge Cases
  a. Company owner tries to configure MFA for a non-existent user → appropriate error
  b. Non-owner user attempts to configure MFA for others → 403 forbidden

5. UI Enhancements
  - Verify that on /admin/mfa/[id] the secret can be toggled visible/invisible and copied

Checklist (manual)
- [ ] MFA Setup: QR code displays after setup call
- [ ] TOTp Verification: valid code enables MFA and returns new JWT when applicable
- [ ] Owner page: MFA status reflects enabled/disabled per user
- [ ] MFA flow on login: once MFA is enabled for a user, login requires MFA
- [ ] Error handling: invalid codes return proper error messages and toasts
- [ ] Security: tokens stored securely on client and server validates MFA state

Notes
- This plan assumes MFA is implemented as a separate step after initial login, and that the company owner panel can configure MFA per user.
- If a simpler MVP is preferred, we can implement a lighter version (owner sets MFA for one user, then test via that user).
