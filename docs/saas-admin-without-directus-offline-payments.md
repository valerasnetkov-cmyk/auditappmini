# ТЗ: переработка панели администратора ресурса без Directus

## Контекст

Проект переводится на модель, где **Directus исключён из архитектуры**.

Вся административная логика SaaS должна жить внутри основного приложения:

```txt
Web SaaS Admin → Backend API → основная БД проекта
```

Нельзя использовать Directus, CMS-синхронизацию, внешнюю backoffice-схему или дублирование компаний/лимитов в отдельной системе.

Администратор ресурса работает в web-панели приложения. Компании, владельцы, тарифы, лимиты, платежи, подписки и уведомления управляются через custom backend.

---

## Цель изменений

Сейчас страница `/saas-admin` содержит слишком много разных сценариев:

- создание компаний;
- создание владельцев;
- создание тарифов;
- настройка лимитов;
- таблицу компаний;
- общую статистику.

При масштабировании эта страница станет перегруженной и неудобной.

Нужно разделить SaaS-админку на несколько логичных разделов:

1. Главный дашборд ресурса.
2. Реестр компаний.
3. Карточка компании.
4. Тарифы и лимиты.
5. Оффлайн-платежи.
6. Уведомления и контроль сроков подписки.

Основная задача — превратить админку ресурса из набора форм в **операционный центр управления SaaS**.

## 1. Главная страница `/saas-admin`

### Назначение

Главная страница не должна быть местом для всех форм и таблиц.

Она должна отвечать на вопросы:

- что происходит с ресурсом сейчас;
- какие компании требуют внимания;
- кто скоро закончит тариф;
- кто просрочил оплату;
- где нет владельца или лимитов;
- сколько денег пришло оффлайн;
- какой расчетный MRR/ARR.

### Убрать с главной страницы

С главной `/saas-admin` нужно убрать:

- большую форму создания компании;
- форму создания владельца;
- форму создания тарифа;
- форму редактирования лимитов;
- большую таблицу всех компаний.

Вместо этого оставить только быстрые действия:

```txt
+ Создать компанию
+ Добавить платеж
+ Создать тариф
```

Кнопки должны вести на отдельные страницы или открывать модальные окна.

### KPI на главной

```txt
Компаний всего
Активные компании
Платные компании
Бесплатные / тестовые компании
MRR
ARR
Оплачено за текущий месяц
Ожидаемые продления
Истекающие тарифы
Просроченные тарифы
Компании без владельца
Компании без лимитов
Компании без активности
```

### Блоки главной страницы

#### 1. Финансы

Показывает:

```txt
MRR
ARR
Оплачено за месяц
Ожидается продлений
Просрочено оплат
Средний чек
```

Важно разделять:

```txt
Фактические платежи = реальные оффлайн-деньги, внесенные вручную.
MRR = расчетная месячная выручка по активным подпискам.
ARR = MRR × 12.
```

Пример:

```txt
Компания оплатила 60 000 ₽ за 6 месяцев.
В платежах: 60 000 ₽.
В MRR: 10 000 ₽.
```


### Поведение после добавления платежа

После сохранения платежа система должна:

1. Создать запись в `company_payments`.
2. Обновить или создать запись в `company_subscriptions`.
3. Обновить `current_period_start`.
4. Обновить `current_period_end`.
5. Проставить `status = active`.
6. Очистить просрочку, если она была.
7. Записать событие в audit log.
8. Пересчитать MRR/ARR.
9. Убрать компанию из списка просроченных, если период актуален.

### Отмена платежа

При отмене платежа:

1. Платеж не удаляется физически.
2. `status` меняется на `cancelled`.
3. В audit log пишется причина отмены.
4. Подписка должна быть пересчитана по последнему активному платежу.

## 8. Предупреждения о завершении тарифа

### Правила

```txt
За 14 дней — мягкое предупреждение админу и владельцу
За 7 дней  — повторное предупреждение
За 3 дня   — срочное предупреждение
За 1 день  — финальное предупреждение
В день окончания — переход в grace
После grace — expired или suspended
```

### Grace period

Для B2B-оффлайн оплат обязательно добавить льготный период.

Рекомендуемое значение:

```txt
7 дней
```

Причина:

- оплата может пройти оффлайн, но еще не быть внесена в систему;
- бухгалтерия клиента может задержать перевод;
- резкая блокировка ухудшит доверие к сервису;
- для MVP важнее сохранить отношения, чем жестко отключать.

## 9. Статусы подписки

```txt
active
expiring
grace
expired
suspended
```

### active

```txt
Тариф действует.
Полный доступ.
```

### expiring

```txt
До окончания тарифа осталось 14 дней или меньше.
Полный доступ.
Показываются предупреждения.
```

### grace

```txt
Срок тарифа закончился.
Доступ временно сохранен.
Показываются заметные предупреждения.
```

### expired

```txt
Срок тарифа истек.
Grace period закончился.
Просмотр истории доступен.
Создание новых сущностей может быть ограничено.
```

### suspended

```txt
Компания ограничена вручную или автоматически.
Доступ только на чтение.
Новые осмотры, загрузка фото, OCR и импорт техники отключены.
```

## 10. Ограничение доступа после окончания тарифа

### Рекомендуемая схема для MVP

```txt
active:
- полный доступ

expiring:
- полный доступ
- предупреждения

grace:
- полный доступ
- заметные предупреждения

expired:
- просмотр данных доступен
- создание новой техники запрещено
- создание новых осмотров запрещено
- загрузка новых фото запрещена

suspended:
- только чтение
- создание осмотров отключено
- создание техники отключено
- загрузка фото отключена
- OCR отключен
- импорт техники отключен
```

### Что нельзя блокировать

Даже при `expired` или `suspended` нельзя скрывать историю:

```txt
Карточки техники
История осмотров
История дефектов
Фото уже проведенных осмотров
Отчеты по старым данным
```

Клиент должен видеть свои данные. Ограничивается только создание новых операций.

## 11. Что видит владелец компании

В панели владельца компании добавить блок тарифа.

### Блок тарифа

```txt
Тариф: Enterprise
Оплачен до: 30.06.2026
Осталось: 6 дней
Техника: 184 / без лимита
Пользователи: 5 / без лимита
OCR: включен
ДТП-модуль: включен
Аналитика: включена
```

### Баннеры

#### За 14 / 7 дней

```txt
Тариф заканчивается через 7 дней. Для продления свяжитесь с администратором сервиса.
```

#### За 3 / 1 день

```txt
Тариф заканчивается через 3 дня. После окончания часть функций может быть ограничена.
```

#### Grace period

```txt
Срок действия тарифа истек. Доступ временно сохранен до 07.07.2026.
```

#### Expired

```txt
Тариф не продлен. Создание новых осмотров и добавление техники временно ограничены.
```

#### Suspended

```txt
Доступ ограничен. Просмотр истории доступен, новые операции временно отключены.
```

## Дополнение: сервисные уведомления для администратора компании

### Контекст

Владелец компании не всегда ежедневно заходит в панель и может пропустить важные сервисные события:

```txt
окончание срока действия тарифа
переход тарифа в grace period
просрочка тарифа
плановые технические работы
обновление системы
изменение условий тарифа
системные инциденты
предупреждения по лимитам
```

Поэтому в панели владельца должна быть возможность назначить одного или нескольких пользователей компании получателями сервисных уведомлений.

Важно разделить роли:

```txt
Администратор компании
```

Но технически это не глобальный `admin`.

## Права администратора компании / manager

Администратор компании — это операционный пользователь внутри компании. Он помогает владельцу следить за работой ресурса, но не управляет SaaS-частью.

Администратор компании может:

```txt

видеть предупреждения о сроке действия тарифа, если он назначен получателем

управлять инспекторами своей компании
создавать инспекторов
редактировать инспекторов
деактивировать инспекторов
смотреть технику, осмотры, дефекты и отчеты своей компании
```

Администратор компании не может:

```txt
назначать других администраторов компании
назначать владельцев
назначать глобальных admin ресурса
менять тариф
менять лимиты
добавлять платежи
отменять платежи
видеть SaaS-admin
видеть другие компании
```

Главное правило:

```txt
Владелец может назначать администраторов компании.
Администратор компании может назначать только инспекторов.
```

## 18. Acceptance criteria

### UI

- `/saas-admin` больше не содержит длинных форм создания компаний/владельцев/тарифов/лимитов.
- `/saas-admin` показывает KPI, истекающие тарифы, health center и последние платежи.
- `/saas-admin/companies` содержит реестр компаний с поиском, фильтрами и действиями.
- `/saas-admin/companies/[id]` показывает карточку компании с вкладками.
- `/saas-admin/payments` позволяет добавить оффлайн-платеж.
- Владелец компании видит статус тарифа и предупреждения.

### Backend

- Все SaaS-admin endpoints доступны только роли `admin`.
- Owner видит только данные своей компании.
- Directus не используется.
- Платежи сохраняются в основной БД.
- Добавление платежа обновляет подписку компании.
- Отмена платежа не удаляет запись физически.
- Подписка корректно переходит между статусами.
- Просроченные компании ограничиваются по правилам.
- История данных не скрывается после просрочки.
- Все действия пишутся в audit log.

### Финансы

- MRR считается отдельно от фактических платежей.
- ARR считается как MRR × 12.
- Оффлайн-платежи показываются по фактической дате оплаты.
- Платеж за несколько месяцев распределяется в MRR помесячно.

### Уведомления

- Система создает уведомления за 14 / 7 / 3 / 1 день до окончания тарифа.
- Владелец компании видит предупреждение в панели.
- Админ ресурса видит истекающие тарифы на главном дашборде.
- Grace period работает после окончания тарифа.

## 21. Важное ограничение

Directus полностью исключён.

Нельзя:

```txt
использовать Directus API

хранить тарифы/лимиты в Directus
показывать Directus status в UI
упоминать Directus в пользовательской панели
```

Вся логика должна быть реализована в:

```txt
backend/
web/
основной базе проекта
```

## 22. Короткий промт для Codex

```txt
Переработай SaaS admin без Directus.

Directus полностью исключен из проекта. Все компании, владельцы, тарифы, лимиты, платежи, подписки и уведомления должны храниться и управляться через custom backend и основную БД проекта.

Нужно разделить текущую страницу /saas-admin:
- /saas-admin — только главный дашборд ресурса: KPI, финансы, истекающие тарифы, health center, активность, последние платежи.
- /saas-admin/companies — реестр компаний с поиском, фильтрами и действиями.
- /saas-admin/companies/new — создание компании.
- /saas-admin/companies/[id] — карточка компании с вкладками: обзор, владелец, тариф и лимиты, платежи, пользователи, активность, уведомления, журнал.
- /saas-admin/plans — управление тарифами.
- /saas-admin/payments — ручной учет оффлайн-платежей.
- /saas-admin/alerts — уведомления и контроль сроков подписок.

Добавь таблицы:
- plans
- company_limits
- company_subscriptions
- company_payments
- company_notifications
- audit_logs

Добавь ручной учет оффлайн-платежей:
- компания
- тариф
- сумма
- валюта
- дата оплаты
- период с
- период по
- метод оплаты
- номер документа
- комментарий

После добавления платежа обновлять подписку компании:
- current_period_start
- current_period_end
- status = active
- last_payment_id
- grace_until

Добавь статусы подписки:
- active
- expiring
- grace
- expired
- suspended

Добавь предупреждения за 14 / 7 / 3 / 1 день до окончания тарифа.
После окончания тарифа переводить компанию в grace на 7 дней.
После grace переводить в expired или suspended.
При expired/suspended не скрывать историю, но ограничивать создание новой техники, новых осмотров, загрузку фото, OCR и импорт.

Admin должен видеть все компании, платежи, подписки и глобальные KPI.
Manager и inspector не должны видеть SaaS-admin.

Обнови API, UI, проверки доступа, audit log и smoke-тесты.
```

## 23. Результат

После внедрения админ ресурса получает не перегруженную страницу с формами, а полноценный SaaS-backoffice:

```txt
/saas-admin — центр решений
/companies — реестр клиентов
/company card — управление клиентом
/payments — учет оффлайн-оплат
/alerts — контроль продлений
```

Такой подход нормально масштабируется от 2 до 200+ компаний и позволяет администратору быстро видеть:

```txt
кто оплатил
кто скоро закончится
кто просрочил
кому нужен владелец
где не настроены лимиты
кого нужно ограничить
кого можно апсейлить
```
