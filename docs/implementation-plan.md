## Этап 8. Проверить сценарии

Минимальные проверки:

- быстрый осмотр полностью проходит;
- плановый осмотр полностью проходит;
- плановый осмотр нельзя завершить без заполнения технических секций;
- дефект ходовой/тормозов/электрики/света создаётся с нужной категорией;
- ДТП-осмотр требует место и время ДТП;
- ДТП-осмотр требует close-up;
- осмотр не завершается без одометра;
- OCR одометра можно исправить вручную;
- номер нельзя ввести кириллицей;
- данные второй компании не видны первой;
- тёмная тема работает на ПК/web;
- тёмная тема работает в mobile;
- режим `system` повторяет тему устройства.

## Этап: региональная архитектура

### 1. Добавить поля региона

```txt
companies.region_code
companies.data_residency
companies.api_cluster_key
companies.storage_cluster_key
companies.ocr_cluster_key
```

### 2. Добавить tenant registry

Минимальный registry:

```txt
slug
public_name
region_code
api_cluster_key
status
```

### 3. Разделить окружения

```txt
staging-ru
staging-intl
production-ru
production-intl
```

EU можно добавить отдельным контуром позже.

### 4. Обновить web/mobile

Web и mobile должны выбирать API по `region_code` компании.

### 5. Обновить OCR

OCR номера и одометра должен вызываться только в регионе компании.

### 6. Обновить storage и backup

Фото и backups должны оставаться внутри региона.

### 7. Добавить проверки

Backend должен возвращать `region_mismatch`, если запрос обращается к компании не из текущего региона.
