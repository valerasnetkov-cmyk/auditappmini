  `inspections.client_inspection_id`, `sync_source`, `completed_at`,
  `odometer_confirmed_at`, `odometer_unavailable_reason`;

- Для client ID созданы частичные уникальные индексы внутри tenant.
  Существующим фотографиям backfill задаёт `captured_at = created_at`.
- `POST /api/inspections` и загрузка фотографий стали идемпотентными.
  Повторная загрузка возвращает существующую запись и удаляет временный
  файловый дубликат.
- `completed` сохранён как признак технической полноты. После завершения
  изменение осмотра, чек-листа, дефектов и фотографий отклоняется с кодом
  `INSPECTION_ALREADY_COMPLETED`. Будущее согласование не смешивается с этим
  статусом.
- Обновление `last_scheduled_inspection` перенесено с создания на успешное
  завершение планового осмотра.

### Readiness и правила завершения

- Добавлен единый `GET /api/inspections/:id/readiness`, используемый backend,
  mobile и web.
- Readiness проверяет обязательные и дополнительные пункты чек-листа,
  сохранённые `null`-ответы, наличие дефекта для каждого ответа «Нет», фото
  каждого дефекта, обязательные категории фотографий и подтверждение пробега.
- Сохранение чек-листа больше не превращает `null` в отрицательный ответ и не
  создаёт для него дефект.
- Для ДТП сохранены прежние 12 обязательных категорий и добавлены `front`,
  `left`, `right`, `rear`, `overview`, `accident_location`. Требуется либо
  подтверждённый пробег с фото одометра, либо причина его недоступности.
- Заблокированное завершение возвращает
  `INSPECTION_COMPLETION_BLOCKED` и структурированный массив `missing` с
  полями `code`, `field` и `label`.

### Mobile

- После создания backend-осмотра активный черновик сохраняется в AsyncStorage,
  а снимки копируются из временного URI в Expo document storage.
- Фотографии сразу видны локально со статусами `local_pending`, `uploading`,
  `uploaded` и `failed`.
- Неблокирующая очередь загружает до двух фото параллельно и повторяется при
  восстановлении приложения в `active`, при старте и вручную.
- Локальные файлы сохраняются до подтверждённой загрузки и очищаются после
  успешного завершения и полной синхронизации. Новый осмотр без сети не
  создаётся, уже созданный переносит потерю соединения.
- Завершение предварительно проверяет readiness и сохраняет структурированные
  причины backend-ошибки для отображения пользователю.

### Watermark, PDF и web

- Sharp создаёт отдельный `watermark.webp`, не изменяя оригинал. Для старых
  фотографий watermark генерируется лениво при первом отчёте.
- Добавлены авторизованные endpoint'ы формирования, чтения metadata и
  скачивания PDF. Незавершённые и чужие tenant-осмотры отклоняются.
- PDF формируется сервером через PDFKit, использует локальный DejaVu Sans для
  кириллицы и QR на авторизованную страницу `/inspections/:id/report`.
  Результат хранит SHA-256; ошибка генерации фиксирует статус `failed`.
- Генерация переведена на уникальный временный файл с публикацией после
  завершения записи и вычисления SHA-256. Параллельные запросы для одного
  tenant/inspection сериализуются и не могут рассинхронизировать файл и hash.
- В `inspection_reports` добавлены `file_size`, `integrity_status` и
  `verified_at`. Получение metadata и скачивание пересчитывают SHA-256 и
  размер; отсутствующий или изменённый PDF получает статус `corrupted`, а
  скачивание отклоняется с `REPORT_INTEGRITY_FAILED`.
- Web показывает причины readiness-блокировки, управляет формированием и
  скачиванием PDF и содержит отдельную авторизованную страницу отчёта.
  Страница показывает результат и время проверки целостности, а повреждённый
  файл нельзя скачать до повторного формирования.

## P1: согласование осмотра

- В `inspections` добавлены повторно безопасные поля `approval_status`,
  `submitted_at/by`, `reviewed_at/by` и `approval_comment`.
- Добавлена tenant-таблица `inspection_approval_history` с неизменяемой
  историей переходов и индексом по компании/осмотру.

  возврата.

  доказательный акт остаётся неизменным; исправленные материалы фиксируются
  новым осмотром.

  manager/owner, решение уведомляет исходного инспектора через существующие
  tenant in-app alerts.
- Web-карточка показывает текущий статус, последний комментарий, историю,
  отправку и доступные руководителю решения.

- Inspection smoke проверяет полный цикл с возвратом, повторной отправкой и

- Chromium E2E проходит путь завершение → отправка → согласование → история →
