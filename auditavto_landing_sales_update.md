# AuditAvto — правки лендинга под продающую подачу

## 1. Цель правок

Обновить публичный лендинг AuditAvto так, чтобы он продавал не просто «аудит техники», а понятную бизнес-ценность для владельцев и руководителей автопарков:

- контроль состояния техники;
- доказательная фотофиксация;
- снижение спорных ситуаций по повреждениям;
- единый стандарт осмотров;
- история дефектов, пробега и ДТП;
- понятный цифровой отчёт вместо хаоса в чатах, таблицах и ручных сообщениях.

Главная конверсия лендинга: заявка на пилот / доступ / демонстрацию сервиса.

Основная аудитория:

- владельцы автопарков;
- транспортные компании;
- сервисные и строительные компании;
- компании с доставкой;
- аренда транспорта и спецтехники;
- руководители, отвечающие за эксплуатацию техники.

---

## 2. Ключевое позиционирование

### Короткая формула

**AuditAvto — цифровой контроль автопарка с фотофиксацией, историей дефектов, пробега и ДТП в одном сервисе.**

### Расширенное позиционирование

AuditAvto помогает владельцам и руководителям автопарков фиксировать состояние техники по единому сценарию: инспектор выбирает автомобиль, проходит чек-лист, делает обязательные фото, отмечает дефекты, подтверждает пробег и формирует отчёт. Все данные сохраняются в истории техники и помогают разбирать спорные ситуации без поиска фото в чатах и таблицах.

### Что НЕ обещать

Не позиционировать проект как систему ремонта, ERP, полноценную TMS или сервис управления исполнителями ремонта. Основная ценность MVP — доказательная фиксация состояния техники, дефектов, пробега, ДТП и истории осмотров.

## 13. FAQ

### Вопрос 1

```txt
Это система ремонта?
```

Ответ:

```txt
Нет. AuditAvto не заменяет ремонтную систему. Его задача — зафиксировать состояние техники, дефекты, пробег, ДТП и историю осмотров.
```

### Вопрос 2

```txt
Можно использовать без QR-кодов?
```

Ответ:

```txt
Да. Основной сценарий — ввод номера техники и подтверждение инспектором.
```

### Вопрос 3

```txt
Подходит для небольшого автопарка?
```

Ответ:

```txt
Да. Сервис можно запустить с небольшой группы техники и масштабировать по мере необходимости.
```

### Вопрос 4

```txt
Кто видит данные?
```

Ответ:

```txt
Доступ зависит от роли: владелец, менеджер, инспектор. Каждый пользователь видит только то, что относится к его компании и задачам.
```

### Вопрос 5

```txt
Можно ли фиксировать ДТП?
```

Ответ:

```txt
Да. Для ДТП предусмотрен отдельный сценарий с фиксацией места, даты, времени, общего плана и крупных планов повреждений.
```

## 16. UX/UI требования

### Общий стиль

- Светлая визуальная система.
- Синий акцент AuditAvto.
- Много воздуха.
- Чёткая иерархия заголовков.
- Минимум декоративных градиентов.
- Карточки с мягкой тенью и радиусами.
- Визуальный фокус на mockup интерфейса и доказательной фотофиксации.

### Что убрать или переработать

- Не оставлять пустые белые блоки под секциями.
- Не размещать форму логина как главный элемент продающего лендинга.
- Вход в систему лучше оставить в шапке отдельной кнопкой `Войти` или компактным блоком ниже.
- Не использовать слишком длинные абзацы на мобильной версии.
- Не делать изображения носителем основного смысла — текст должен индексироваться и быть доступным.

### CTA-логика

CTA должен повторяться минимум 3 раза:

1. Hero.
2. После блока «Как работает осмотр».
3. После оффера пилотного запуска / в финале.

## 17. Технические требования для Codex

### Где вносить правки

Определить текущий файл публичной главной страницы. Возможные варианты:

```txt
web/src/app/page.tsx
web/src/app/(marketing)/page.tsx
web/src/app/landing/page.tsx
```

Если текущая главная страница смешивает публичный лендинг и форму входа, разделить:

```txt
/          -> публичный продающий лендинг
/login     -> вход в систему
```

### CSS

Стили должны быть вынесены во внешний CSS-файл. Не использовать inline styles.

Возможные файлы:

```txt
web/src/app/landing.css
web/src/styles/landing.css
web/src/styles/marketing.css
```

### Компонентная структура

Рекомендуется разбить лендинг на компоненты:

```txt
web/src/components/landing/LandingHero.tsx
web/src/components/landing/LandingProblem.tsx
web/src/components/landing/LandingSolution.tsx
web/src/components/landing/LandingHowItWorks.tsx
web/src/components/landing/LandingDefects.tsx
web/src/components/landing/LandingMileageAccident.tsx
web/src/components/landing/LandingRoles.tsx
web/src/components/landing/LandingBusinessBenefits.tsx
web/src/components/landing/LandingOffer.tsx
web/src/components/landing/LandingFaq.tsx
web/src/components/landing/LandingFinalCta.tsx
```

Если проект пока небольшой, допускается оставить секции в одном файле, но не раздувать компонент выше разумного размера. При росте страницы — декомпозировать.

Пример структуры:


## 18. Acceptance Criteria

Правки считаются выполненными, если:

- первый экран сразу объясняет бизнес-ценность для владельца автопарка;
- на странице явно раскрыты боли: чаты, таблицы, потерянные фото, спорные дефекты, ручной пробег;
- есть понятная связка «боль → решение → выгода»;
- CTA ведут к заявке, доступу или демо;
- блоки `Как работает осмотр`, `Фотофиксация дефектов`, `Пробег и ДТП`, `Безопасность и роли` обновлены продающим текстом;
- пустые визуальные карточки заменены изображениями, mockup или смысловыми блоками;
- форма входа не конкурирует с основным CTA лендинга;
- страница корректно выглядит на мобильном, планшете и десктопе;
- стили вынесены во внешний CSS;
- нет inline-style в TSX/JSX;
- изображения оптимизированы и не ломают загрузку страницы;
- `npm --prefix web run lint` проходит без ошибок;
- `npm --prefix web run build` проходит без ошибок.
